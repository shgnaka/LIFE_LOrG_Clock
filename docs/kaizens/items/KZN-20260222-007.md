Title: Replace error()-based failure paths with typed parser/repository failures
Category: data
Status: Done
Priority: P1
Impact: High
Effort: M
Updated: 2026-02-22

## Problem
Core parser/repository/domain paths still rely on `error(...)` and generic runtime exceptions for recoverable conditions. This increases crash risk and makes failure handling inconsistent.

## Evidence
- app/src/main/java/com/example/orgclock/parser/OrgParser.kt:141
- app/src/main/java/com/example/orgclock/parser/OrgParser.kt:210
- app/src/main/java/com/example/orgclock/data/SafOrgRepository.kt:34
- app/src/main/java/com/example/orgclock/data/SafOrgRepository.kt:43
- app/src/main/java/com/example/orgclock/domain/ClockService.kt:390

## Hardcoded/Smell
- `error(...)` is used for state/validation failures that can occur from user data.
- Failure semantics are conveyed by free-form message strings.
- Error mapping is spread and not type-driven.

## Proposed Change
- Replace `error(...)` usages in parser/repository/domain with explicit typed failures:
  - `IllegalArgumentException` for validation issues
  - `SaveResult.ValidationError` / `SaveResult.IoError` where appropriate
  - domain-specific exception types where needed
- Standardize conversion into user-facing failures at service boundary.
- Ensure no user-triggered path relies on process-terminating failure primitives.

## Acceptance Criteria
- `kotlin.error(...)` is not used in `OrgParser`, `SafOrgRepository`, or `ClockService`.
- Missing heading/clock scenarios return structured failures, not fatal runtime errors.
- Existing UI flows surface clear failure statuses instead of crashing.

## Test Plan
- Add parser tests for missing heading and malformed clock cases that assert failure results.
- Add service tests for conflict/validation branches to verify mapped error types.
- Run `./gradlew :app:testDebugUnitTest`.

## Out of Scope
- Replacing all exceptions across the entire app with a global error framework.

## Dependencies
- None.
