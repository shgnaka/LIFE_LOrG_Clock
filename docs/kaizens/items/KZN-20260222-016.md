Title: Make OrgRepository contract strict by removing default stub implementations
Category: tests-quality
Status: Ready
Priority: P2
Impact: Medium
Effort: S
Updated: 2026-02-22

## Problem
`OrgRepository` defines default stub implementations for core methods. Missing overrides can compile and fail only at runtime, weakening contract safety and hiding implementation gaps until execution.

## Evidence
- app/src/main/java/com/example/orgclock/data/RepositoryModels.kt:34
- app/src/main/java/com/example/orgclock/data/RepositoryModels.kt:39
- app/src/main/java/com/example/orgclock/data/RepositoryModels.kt:48
- app/src/main/java/com/example/orgclock/domain/ClockService.kt:74

## Hardcoded/Smell
- Interface methods return placeholder failures instead of enforcing required implementation at compile time.
- Runtime-only guard rails (`"is not implemented"`) can leak into production paths if wiring is incomplete.

## Proposed Change
- Remove default method bodies for `listOrgFiles`, `loadFile`, and `saveFile` from `OrgRepository`.
- Make all repository implementations and test doubles explicitly implement the full interface.
- Replace any reliance on default stubs with explicit fake behavior in tests.
- Keep method signatures and existing call sites unchanged.

## Acceptance Criteria
- [ ] `OrgRepository` no longer contains default stub bodies for core file methods.
- [ ] No `"is not implemented"` fallback strings remain for those methods in production source.
- [ ] All repository implementations/test doubles compile with explicit implementations.
- [ ] Unit tests pass after contract tightening.

## Test Plan
- Update affected test doubles in `ClockServiceTest`, `ClockInScannerTest`, and ViewModel tests as needed.
- Run `./gradlew :app:testDebugUnitTest`.
- Confirm no runtime paths depend on removed default stubs.

## Out of Scope
- Adding new repository capabilities.
- Refactoring repository consumers beyond required compile fixes.

## Dependencies
- None.
