## Problem
Notification service sync is currently tied to broad UI state changes, including heading list updates. Clock operations and optimistic UI updates can trigger frequent redundant sync calls, causing unnecessary service churn and extra background work.

## Evidence
- app/src/main/java/com/example/orgclock/ui/app/OrgClockRoute.kt:115
- app/src/main/java/com/example/orgclock/ui/app/OrgClockRoute.kt:119
- app/src/main/java/com/example/orgclock/ui/viewmodel/OrgClockViewModel.kt:397
- app/src/main/java/com/example/orgclock/ui/viewmodel/OrgClockViewModel.kt:718

## Hardcoded/Smell
- Side-effect trigger key includes `state.headings`, which is updated for many UI-only reasons.
- Notification sync policy is not separated from transient heading rendering state.

## Proposed Change
- Narrow sync trigger inputs to notification-relevant state (`notificationEnabled`, `notificationDisplayMode`, `rootUri`, and open-clock footprint).
- Remove `headings` from the sync `LaunchedEffect` key set.
- Add a short debounce/coalescing step to collapse back-to-back state transitions into one sync call.
- Keep stop behavior intact when notifications are disabled or root URI is missing.

## Acceptance Criteria
- [ ] Updating heading UI state alone does not call `syncNotificationService`.
- [ ] Notification mode toggle still triggers exactly one sync update per user action.
- [ ] Start/stop/cancel operations still propagate notification updates via notification-relevant state changes.
- [ ] No regression in service stop path when notifications are disabled.

## Test Plan
- Add route-level test with fake dependencies that count `syncNotificationService` calls under heading-only state changes.
- Add test for notification setting change to verify expected sync call count.
- Run `./gradlew :app:testDebugUnitTest`.
- Manual smoke: rapid clock start/stop should not cause repeated redundant sync bursts.

## Out of Scope
- Redesigning notification settings UI.
- Changing notification content/layout.

## Dependencies
