## Problem
Notification permission checks are duplicated in both `MainActivity` and `ClockInNotificationService`. Duplicated platform checks can drift and create inconsistent enable/disable behavior across entry points.

## Evidence
- app/src/main/java/com/example/orgclock/notification/NotificationPermissionChecker.kt:10
- app/src/main/java/com/example/orgclock/notification/NotificationPermissionChecker.kt:14
- app/src/main/java/com/example/orgclock/MainActivity.kt:87
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:60
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:97
- app/src/test/java/com/example/orgclock/notification/DefaultNotificationPermissionCheckerTest.kt:9

## Hardcoded/Smell
- Same permission policy logic is implemented in two classes.
- API-level branching and notification-manager checks are repeated.
- Security/permission behavior has no single source of truth.

## Proposed Change
- Extract shared permission policy into a single helper/component (for example `NotificationPermissionChecker`).
- Route both Activity and Service through the shared checker.
- Keep behavior unchanged for pre-Android 13 and Android 13+.

## Acceptance Criteria
- [x] There is exactly one production implementation of notification permission policy.
- [x] `MainActivity` and `ClockInNotificationService` both consume the shared implementation.
- [x] Permission behavior remains equivalent for:
  - Android < 33 (notification manager enabled/disabled)
  - Android >= 33 (runtime permission + notification manager)
