ID: KZN-20260222-006
Title: Unify notification permission checks across Activity and Service
Category: security-privacy
Status: Ready
Priority: P2
Impact: Medium
Effort: S
Updated: 2026-02-22

## Problem
Notification permission checks are duplicated in both `MainActivity` and `ClockInNotificationService`. Duplicated platform checks can drift and create inconsistent enable/disable behavior across entry points.

## Evidence
- app/src/main/java/com/example/orgclock/MainActivity.kt:101
- app/src/main/java/com/example/orgclock/MainActivity.kt:110
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:249
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:258

## Hardcoded/Smell
- Same permission policy logic is implemented in two classes.
- API-level branching and notification-manager checks are repeated.
- Security/permission behavior has no single source of truth.

## Proposed Change
- Extract shared permission policy into a single helper/component (for example `NotificationPermissionChecker`).
- Route both Activity and Service through the shared checker.
- Keep behavior unchanged for pre-Android 13 and Android 13+.

## Acceptance Criteria
- There is exactly one production implementation of notification permission policy.
- `MainActivity` and `ClockInNotificationService` both consume the shared implementation.
- Permission behavior remains equivalent for:
  - Android < 33 (notification manager enabled/disabled)
  - Android >= 33 (runtime permission + notification manager)

## Test Plan
- Add unit tests for shared checker covering API-level and permission combinations.
- Run `./gradlew :app:testDebugUnitTest`.
- Manual smoke on Android 13+ device/emulator: toggle permission and verify UI/service behavior remains consistent.

## Out of Scope
- Changing permission request UX flow or wording.
- Adding new notification channels.

## Dependencies
- None.
