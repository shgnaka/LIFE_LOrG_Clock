## Problem
Benchmark waits use duplicated literal timeout values (`5_000`) across multiple scenarios. This scatters benchmark synchronization policy and makes test tuning error-prone.

## Evidence
- benchmark/src/androidTest/java/com/example/orgclock/benchmark/StartupAndScrollBenchmark.kt:37
- benchmark/src/androidTest/java/com/example/orgclock/benchmark/StartupAndScrollBenchmark.kt:52
- benchmark/src/androidTest/java/com/example/orgclock/benchmark/StartupAndScrollBenchmark.kt:72

## Hardcoded/Smell
- The same timeout literal is repeated in multiple benchmark methods.
- Wait policy cannot be changed centrally for all benchmark scenarios.

## Proposed Change
- Introduce a single benchmark timeout constant (for example `UI_WAIT_TIMEOUT_MS`).
- Replace the three inline `5_000` values with the shared constant.
- Keep timeout value unchanged to avoid behavior drift.

## Acceptance Criteria
- All `UiDevice.wait` calls in `StartupAndScrollBenchmark` use one shared timeout constant.
- No inline `5_000` timeout literal remains in the file.
- Benchmark behavior is unchanged under default constant value.

## Test Plan
- Run benchmark module tests on device/emulator and confirm no compile/runtime regression.
- Confirm benchmark methods (`coldStartup`, `filePickerFrameTiming`, `headingListScrollFrameTiming`) still execute.

## Out of Scope
- Changing benchmark metrics, iterations, or startup mode configuration.

## Dependencies
- None.
