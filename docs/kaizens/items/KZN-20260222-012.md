Title: Eliminate O(n^2) file lookup during notification scans
Category: data
Status: Ready
Priority: P1
Impact: High
Effort: M
Updated: 2026-02-22

## Problem
Notification scanning loads each file one-by-one. `loadFile` currently resolves file IDs by re-listing all files and linearly searching for URI matches each time, turning one scan into repeated directory traversal. As file count grows, scan latency and battery cost increase significantly.

## Evidence
- app/src/main/java/com/example/orgclock/notification/ClockInScanner.kt:25
- app/src/main/java/com/example/orgclock/notification/ClockInScanner.kt:26
- app/src/main/java/com/example/orgclock/data/SafOrgRepository.kt:117
- app/src/main/java/com/example/orgclock/data/SafOrgRepository.kt:251

## Hardcoded/Smell
- `findFileById` performs `rootDoc.listFiles()` + linear scan for every `loadFile`/`saveFile`.
- Scanner iterates all files, so repeated full-directory enumeration compounds to O(n^2).

## Proposed Change
- Replace `findFileById` linear search with direct URI resolution (`DocumentFile.fromSingleUri`) and validity checks.
- Refactor `loadFile` and `saveFile` to use the same direct resolver path and remove repeated `rootDoc.listFiles()` traversal.
- Keep existing validation semantics (`File not found`, invalid name, hash conflict) unchanged.
- Add a focused repository micro-benchmark or timing harness to measure scan-path improvement.

## Acceptance Criteria
- [ ] `SafOrgRepository.loadFile` and `saveFile` no longer enumerate all files for each file-ID lookup.
- [ ] Notification scanning works with unchanged functional behavior across start/stop/history flows.
- [ ] Measured notification scan time improves on directories with many `.org` files.
- [ ] No regression in conflict detection (`expectedHash` mismatch path).

## Test Plan
- Add/extend repository tests around file ID resolution and error paths with direct URI lookup.
- Run scanner tests (`ClockInScannerTest`) and domain tests that depend on `loadFile`/`saveFile` behavior.
- Run `./gradlew :app:testDebugUnitTest`.
- Manual benchmark on a directory with many `.org` files and compare scan duration before/after.

## Out of Scope
- Building a persistent file index database.
- Changing file naming conventions or backup behavior.

## Dependencies
- None.
