## Problem
The App Distribution upload wait loop uses embedded retry/sleep literals. Timeout policy intent is hidden in shell code and is not easy to tune per workflow run.

## Evidence
- .github/workflows/distribute-android.yml:163
- .github/workflows/distribute-android.yml:187
- .github/workflows/distribute-android.yml:190

## Hardcoded/Smell
- `seq 1 60` and `sleep 2` are embedded literal values.
- Poll policy cannot be observed from logs without inspecting workflow source.

## Proposed Change
- Replace loop literals with env vars:
  - `UPLOAD_POLL_MAX_ATTEMPTS` (default `60`)
  - `UPLOAD_POLL_SLEEP_SEC` (default `2`)
- Compute effective timeout in logs at runtime (`attempts * sleep`).
- Keep default behavior equivalent to current 120-second wait budget.

## Acceptance Criteria
- Workflow behavior is identical under default values.
- Polling policy can be changed without touching loop body logic.
- Workflow logs print resolved attempts and sleep seconds before loop starts.
- No hardcoded numeric literals remain in loop header/sleep statement.

## Test Plan
- Run YAML validation/lint in CI (or local syntax check if available).
- Trigger one manual workflow dispatch and confirm logs show effective poll config.
- Verify successful run with defaults and no regression in release distribution path.

## Out of Scope
- Changing authentication architecture.

## Dependencies
- None.
