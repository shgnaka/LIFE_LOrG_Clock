## Problem
Minute-step and ticker timing behavior is implemented with repeated literals in both UI and ViewModel. This risks drift between edit rounding rules and selectable minute options.

## Evidence
- app/src/main/java/com/example/orgclock/ui/app/OrgClockScreen.kt:810
- app/src/main/java/com/example/orgclock/ui/app/OrgClockScreen.kt:650
- app/src/main/java/com/example/orgclock/ui/app/OrgClockScreen.kt:651
- app/src/main/java/com/example/orgclock/ui/viewmodel/OrgClockViewModel.kt:600

## Hardcoded/Smell
- Minute step (`5`) is duplicated in UI option generation and ViewModel normalization.
- Running panel tick interval (`60_000L`) is inlined in elapsed-time loop logic.
- Time edit/ticker policy lacks a single source of truth.

## Proposed Change
- Introduce shared UI-time config constants (for example `MINUTE_STEP`, `RUNNING_PANEL_TICK_MS`).
- Generate minute dropdown options from shared minute-step constant.
- Refactor minute normalization helper to use shared constant and keep boundary behavior.

## Acceptance Criteria
- Minute dropdown options and minute normalization both derive from the same step constant.
- Running panel update loop uses a named tick constant/helper rather than inline `60_000L`.
- Existing edit behavior remains equivalent (including minute rounding edge cases).

## Test Plan
- Add/extend unit tests for minute normalization boundaries (`0`, `2`, `58`, `59`).
- Run `./gradlew :app:testDebugUnitTest`.
- Manual smoke: edit closed clock times and confirm selectable/rounded minutes remain consistent.

## Out of Scope
- Introducing second-level editing precision.
- Redesigning clock edit dialog UX.

## Dependencies
- None.
