## Problem
`MainActivity` directly instantiates repository/service/scanner and wires many lambdas into `OrgClockRoute`. This couples app wiring, time/zone acquisition, and UI entrypoint lifecycle in one class, making dependency replacement and wiring tests difficult.

## Hardcoded/Smell
- Composition root logic is tightly coupled to activity lifecycle class.
- Dependency graph is manually assembled in one large block with no reusable container.
- `ZonedDateTime.now()` and `ZoneId.systemDefault()` are pulled directly at call sites in the same block.

## Proposed Change
- Introduce a small app container (`AppGraph`) that creates and exposes app-level dependencies.
- Move repository/service/scanner creation and settings access wrappers into the container layer.
- Keep `MainActivity` focused on lifecycle + `setContent` and consume container-provided interfaces.
- Prepare extension points for shared clock/zone provider usage.

## Acceptance Criteria
- `MainActivity` no longer directly instantiates `SafOrgRepository`, `ClockService`, or `ClockInScanner`.
- `OrgClockRoute` receives dependencies from a graph/container object, not ad-hoc per-lambda wiring.
- A fake graph/container can be supplied in tests without editing production activity code.
- Existing app startup behavior and screen transitions remain unchanged.

## Test Plan
- Run `./gradlew :app:assembleDebug`.
- Run `./gradlew :app:testDebugUnitTest`.
- Add a small wiring test (or constructor test) validating graph can construct route dependencies.

## Out of Scope
- Full DI framework migration.
