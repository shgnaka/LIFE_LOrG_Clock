ID: KZN-20260222-002
Title: Decouple MainActivity composition root from concrete implementations
Category: domain
Status: Open
Priority: P2
Impact: High
Effort: M
Updated: 2026-02-22

## Problem
`MainActivity` instantiates concrete repository/services directly and passes many lambdas, which reduces replaceability and testability of wiring.

## Evidence
- app/src/main/java/com/example/orgclock/MainActivity.kt:31
- app/src/main/java/com/example/orgclock/MainActivity.kt:32
- app/src/main/java/com/example/orgclock/MainActivity.kt:39

## Hardcoded/Smell
- Composition root logic is tightly coupled to activity lifecycle class.
- Dependency graph is manually assembled in one large block.

## Proposed Change
- Introduce a small app container (`AppGraph` or equivalent) with interfaces for use cases.
- `MainActivity` should request dependencies from the container and remain thin.

## Acceptance Criteria
- `MainActivity` no longer creates `SafOrgRepository`/`ClockService`/`ClockInScanner` directly.
- Dependency wiring can be swapped in tests without editing activity source.
- Existing UI behavior is preserved.

## Test Plan
- Compile test for app module.
- Existing ViewModel tests pass with no behavior regression.

## Out of Scope
- Full DI framework migration.

## Dependencies
- Coordinate with any ongoing architecture refactor PR.
