## Problem
Notification polling interval and list display limits are hardcoded in service constants, making tuning and testing expensive.

## Evidence
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:291
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:292
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:187

## Hardcoded/Smell
- Hardcoded values: `SCAN_INTERVAL_MS = 5 * 60 * 1000L`, `MAX_LINES = 6`.
- Behavior-level constants are embedded in service implementation.

## Proposed Change
- Introduce a `NotificationConfig` interface or data class injected into `ClockInNotificationService`.
- Move default values to one central configuration file.
- Make values overrideable in tests.

## Acceptance Criteria
- Service no longer hardcodes polling interval or max inbox lines.
- Unit tests can set small interval / max lines deterministically.
- Existing runtime behavior remains equivalent with default config.

## Test Plan
- Unit test: service builds inbox notification respecting injected max lines.
- Manual: verify notification still refreshes and renders expected lines.

## Out of Scope
- Redesigning notification UX wording.

## Dependencies
- None.
