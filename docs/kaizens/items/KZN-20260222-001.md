## Problem
Notification polling interval and inbox display limits are embedded as literals in the foreground service. This makes runtime tuning and deterministic notification tests costly because values can only be changed by editing service code.

## Evidence
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:93
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:291
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:292
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:194
- app/src/main/java/com/example/orgclock/notification/ClockInNotificationService.kt:203

## Hardcoded/Smell
- Hardcoded behavior values: `SCAN_INTERVAL_MS = 5 * 60 * 1000L`, `MAX_LINES = 6`.
- The same class owns both orchestration logic and tuning policy.

## Proposed Change
- Introduce `NotificationServiceConfig` (or equivalent) with `scanIntervalMs` and `maxLines`.
- Inject config into `ClockInNotificationService` via composition root.
- Keep default values equivalent to current behavior and expose test overrides.

## Acceptance Criteria
- `ClockInNotificationService` no longer defines `SCAN_INTERVAL_MS` or `MAX_LINES` literals.
- Foreground loop delay and inbox truncation both read from injected config.
- With default config values, notification behavior remains functionally identical.
- At least one unit test verifies truncation behavior with `maxLines = 2`.

## Test Plan
- Add/extend unit tests to verify inbox lines and `"...more"` behavior with custom `maxLines`.
- Add/extend unit tests to verify periodic refresh delay uses configurable interval.
- Manual smoke: enable notification in app settings and confirm periodic refresh still occurs.

## Out of Scope
- Redesigning notification copy or notification channel behavior.

## Dependencies
- None.
