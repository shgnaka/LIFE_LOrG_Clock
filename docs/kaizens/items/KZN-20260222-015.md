Title: Split oversized OrgClockScreen and ViewModel action dispatcher
Category: ui

Updated: 2026-02-22

## Problem
UI and state orchestration are concentrated in oversized files. `OrgClockScreen` combines root/file/heading/settings screens with multiple dialogs, and `OrgClockViewModel.onAction` contains a very large dispatch branch. This increases review cost, regression risk, and change conflicts.

## Evidence
- app/src/main/java/com/example/orgclock/ui/app/OrgClockScreen.kt:139
- app/src/main/java/com/example/orgclock/ui/app/OrgClockScreen.kt:206
- app/src/main/java/com/example/orgclock/ui/viewmodel/OrgClockViewModel.kt:69
- app/src/main/java/com/example/orgclock/ui/viewmodel/OrgClockViewModel.kt:392

## Hardcoded/Smell
- Multiple dialog implementations and screen sections are embedded in one large composable file.
- Action routing and feature logic are tightly coupled in one monolithic `when` block.

## Proposed Change
- Extract dialog composables (`history`, `delete`, `edit`, `create heading`) into dedicated files/components under `ui/app`.
- Keep `OrgClockScreen` focused on top-level routing and screen composition.
- Split ViewModel action handling into feature-focused handler methods (navigation, clock ops, history/edit/delete, heading creation, notifications).
- Preserve `OrgClockUiAction` and `OrgClockUiState` contracts to avoid broad API churn.

## Acceptance Criteria
- [ ] `OrgClockScreen.kt` no longer contains all dialog implementations inline.
- [ ] `OrgClockViewModel.onAction` delegates to feature-scoped handlers rather than inlining all logic.
- [ ] Existing behavior for start/stop/cancel, history edit/delete, heading creation, and notification settings remains unchanged.
- [ ] Existing test suites continue to pass without behavior regressions.

## Test Plan
- Run `./gradlew :app:testDebugUnitTest`.
- Run `./gradlew :app:connectedDebugAndroidTest` for basic UI checks where available.
- Manual smoke for root setup, file selection, clock start/stop/cancel, history edit/delete, and settings navigation.

## Out of Scope
- Visual redesign or new navigation architecture.
- DI framework migration.

## Dependencies
- None.
